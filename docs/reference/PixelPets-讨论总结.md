# PixelPets — AI 订阅额度像素宠物伴侣

## 讨论总结

### 一、起点：寻找「小却幸福」的工具

用户希望做一个「几天能做出来的小工具」，不是社交平台或电商网站，而是能**极大提升幸福感**的东西——那种「没有它也能活，但有了之后很舒服」的工具。

我从 GitHub 的 `awesome-cli-apps` 和 `awesome-mac` 中筛选了 16 个候选创意，分为四大类：
- **A. 效率型**：全局快捷词、剪贴板时光机、悬停预读器、窗口暂存区
- **B. 情绪型**：菜单栏数字植物、一键雨声、每日宇宙来信、成就解锁系统
- **C. 开发者型**：项目上下文切换器、API 便签本、Git 情绪日历、日志彩色流
- **D. 生活管理型**：微记账、身体节律钟、物品定位器、关系维护提醒

### 二、创意聚焦：像素宠物 × AI 订阅额度

用户对 **B1「菜单栏数字植物」** 产生强烈兴趣，并注入了一个极其巧妙的创意：

> **用 AI 订阅额度（Claude / Codex / Gemini）的消耗量来驱动宠物生长。**

这个创意的魅力在于：
- **实用性**：一眼看到额度还剩多少，是否需要切换模型
- **情绪价值**：把冰冷的数字变成有生命的陪伴
- **仪式感**：额度刷新时宠物「打瞌睡」，成了一种温柔的提醒

### 三、核心设计决策与原因

#### 决策 1：混合生态（植物 + 动物）

| 生物类型 | 生长特性 | 选择原因 |
|---------|---------|---------|
| **植物** | 有生长上限，达到成熟后停滞，但可触发开花/结果等奖励动画 | 符合植物真实特性，给用户「养成」的成就感 |
| **动物** | 无生长上限，持续活跃，有多种随机行为 | 动物本就该活蹦乱跳，持续提供情绪价值 |

**决策原因**：植物和动物的生物学特性天然适合不同的反馈模式，混合起来既有「里程碑感」又有「持续陪伴感」。

---

#### 决策 2：生长与额度消耗的关系

**规则**：
- **消耗越多 → 宠物越茂盛/活跃**（正向激励）
- **5 小时额度 / 每周额度 未使用 → 宠物打瞌睡**（提示额度已刷新，可以用起来了）

**决策原因**：
- 打瞌睡不是惩罚，而是一种**温柔的信号**——「你的额度睡醒了，该用起来了」
- 避免了「枯萎死亡」的负面体验，保持了工具的治愈属性
- 直接解决了用户的真实痛点：经常忘了额度什么时候刷新

---

#### 决策 3：技术栈 —— Swift 原生轻量方案

| 候选方案 | 结论 | 原因 |
|---------|------|------|
| Tauri 全复用 | ❌ 放弃 | 太重，超出「几天做出来」的范围 |
| 无代理纯文件监控 | ❌ 放弃 | 各 CLI 日志格式不统一，数据不可靠 |
| 内嵌 cc-switch Rust 代理 | ❌ 放弃 | 双语言维护成本高 |
| **Swift + SwiftUI + SwiftNIO 代理** | ✅ **选定** | 原生 macOS 体验最佳，菜单栏动画最流畅，代理逻辑可控 |

**核心架构**：
```
SwiftUI 菜单栏 App
  ├── NWListener Hook 接收器 (localhost:15799)
  │     ├── 拦截 Claude / Codex / Gemini / OpenCode API 请求
  │     ├── 解析响应 JSON 中的 token 用量
  │     ├── 计算费用（内置模型价格表）
  │     └── 写入 SQLite 数据库
  ├── 菜单栏图标：逐帧像素动画（SpriteKit / SwiftUI Canvas）
  └── 下拉面板：大宠物 + 用量进度条 + 额度刷新倒计时
```

---

#### 决策 4：数据获取 —— 本地代理（参考 cc-switch）

**方案**：启动本地 HTTP 代理，自动修改各 CLI 的 base URL

**操作流程**：
1. App 首次启动 → 扫描 `~/.claude/`、`~/.codex/` 等目录，检测已安装的 CLI
2. 弹出面板：「检测到 Claude Code 和 Gemini CLI，是否启用追踪？」
3. 用户点击「启用」→ App 自动修改环境变量/配置文件：
   - Claude Code：Node.js Hook POST 到 `http://127.0.0.1:15799/state`
   - Codex：`~/.codex/config.toml` 中的 `base_url`
   - Gemini CLI：`GOOGLE_GEMINI_BASE_URL`
   - OpenCode：对应配置
4. App 启动代理，开始记录用量

**参考来源**：cc-switch（https://github.com/farion1231/cc-switch）
cc-switch 的核心技术方案：
- 本地代理绑定 `127.0.0.1:15721`
- 修改 CLI base URL 使流量经过代理
- 代理转发到官方 API，同时解析响应 JSON 提取 token 用量
- 费用计算：内置 `model_pricing` 表，支持 input/output/cache_read/cache_creation 四级计费
- 数据存储：SQLite（`proxy_request_logs`、`usage_daily_rollups`）

**决策原因**：不依赖外部日志格式，数据最准确，且用户有 cc-switch 的使用经验，对「一键连接」的体验有明确预期。

---

#### 决策 5：显示方式 —— 菜单栏图标动画 + 下拉详情

| 层级 | 内容 | 尺寸 |
|------|------|------|
| **菜单栏图标** | 逐帧循环像素宠物动画（idle 状态） | 16×16 px |
| **点击展开面板** | 大宠物 + 各模型用量进度条 + 额度刷新倒计时 | 300×400 px 左右 |

**决策原因**：
- 图标本身有生命 → 即使不打开面板，也能感受到陪伴
- 面板提供详细信息 → 需要看用量时，信息一目了然

---

#### 决策 6：宠物状态机

| 状态 | 触发条件 | 植物表现 | 动物表现 |
|------|---------|---------|---------|
| **打瞌睡** | 额度刷新后，5h/weekly 额度未开始消耗 | 叶子低垂，冒 Zzz | 蜷缩，尾巴盖住鼻子 |
| **苏醒** | 检测到第一个 API 请求 | 伸懒腰，抖落露水 | 伸懒腰，竖耳朵 |
| **生长中** | 持续使用，消耗额度 | 长叶子 → 长高大 | 摇尾巴 → 跑跳 → 翻肚皮 |
| **成熟态** | 植物达到最大生长阶段 | 大树，不再长高 | — |
| **开花/结果** | 成熟后继续消耗，触发奖励动画 | 开花 → 结果 | — |
| **活跃多样** | 动物持续使用 | — | 飞、啄、唱歌、洗澡等随机行为 |

**决策原因**：
- 植物的「停滞」符合真实生物学
- 动物的「持续活跃」符合人们对宠物的预期
- 「打瞌睡」作为额度刷新的信号，是最温柔的提醒方式

---

#### 决策 7：多宠物领养

- 每个 AI 订阅对应一个宠物槽位
- **默认随机**分配种类（植物或动物 + 具体类型）
- 用户可修改：点击宠物 → 弹出「换宠物」面板 → 选择新种类

**决策原因**：
- 随机分配降低决策成本，开箱即用
- 可修改给予用户掌控感，避免「我不喜欢默认宠物」的抵触
